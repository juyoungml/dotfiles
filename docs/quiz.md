# Dotfiles MCQ Quiz

Run `dots quiz` for interactive grading, or pipe/open this file to study. Each
question has one best answer.

## Learning Outcomes

After finishing this quiz, you should be able to:

- install, update, inspect, and safely re-run the dotfiles setup;
- explain why shell startup stays small and where each shell behavior belongs;
- use the daily CLI helpers for navigation, Git, uv, secrets, tmux, Slurm, and VAST;
- keep public dotfiles separate from private machine state, secrets, and raw agent memory;
- choose safer workflows for cluster jobs, large data operations, and productivity automation.

## Mental Model

Q: What is the main design goal of this dotfiles repo?
A) A minimal, fast, inspectable setup that is easy to install and remove
B) A feature-rich shell framework with many plugins
C) A private backup of every local config directory
D) A cluster provisioning system for GPU drivers
Answer: A
Why: The repo favors small readable files, cheap startup, and reversible install behavior.

Q: What should shell startup avoid?
A) Cheap environment variables
B) Network calls, heavy scans, secret loading, and project-specific probes

D) Loading aliases

Why: Startup must stay synchronous, deterministic, and cheap.

Q: When should a new alias or tool be added?
A) Whenever it is interesting
B) Whenever a package manager recommends it

Answer: C
Why: The repo optimizes for useful surface area, not a large command collection.

Q: What belongs in public dotfiles?
A) Raw tokens and private memory
B) Session histories and auth databases
C) Every local configuration file
D) Safe defaults, examples, reviewed instructions, and reusable skills

Why: Public files must stay safe to publish and portable across machines.
